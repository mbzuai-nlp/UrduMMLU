Annotator guidelines

What to do, how to do it, and what to flag. Skim once before you start; come back if anything feels ambiguous.

What we're doing

You're verifying answers for the Urdu MMLU benchmark — a multiple-choice question set covering Pakistani academic subjects in Urdu. Every question has been pre-processed; your job is to pick the single best answer, and to flag questions that have problems (wrong subdomain, typos, ambiguity, OCR errors, etc.).

Each question is annotated by two different annotators (you and one teammate, independently). Where you agree, the answer is confirmed. Where you disagree, the admin reviews the question.

Using the interface

  • Log in with the handle the admin assigned you.
  • You'll see a list of batches (about 50 MCQs each).
  • Open one batch at a time; your progress is saved automatically in your browser. You can stop and come back any time.
  • Click an option to record your answer. Clicking does not advance — use → or the next button when you're ready to move on.
  • When a batch is done (or whenever you want to send progress to the admin), click Export and email/Slack the JSON file.

Keyboard shortcuts: 1–5 pick A/B/C/D/E (or unsure) · ←/→ navigate · F toggle flag · E toggle edit mode.

Picking the answer

  • Pick the single best answer. If multiple options seem correct, choose the one that's most precise or most directly answers the question.
  • Look it up if you're not sure. A quick Google / Wikipedia check is encouraged for fact-based questions (dates, authors, capitals, scientific terms, historical events). The benchmark needs correct answers, not gut-feel answers.
  • If a search doesn't resolve it within ~30 seconds, mark unsure and move on. Confident guesses are worse than honest unsure-marks.
  • Aim for 15–30 seconds per question. If you find yourself on one question for more than a minute even after a quick search, mark unsure.
Googling is fine — preferred, even. Open a tab, search the question or option text, verify, then pick. A few minutes of checking per batch is well spent.

When to use "unsure / skip"

  • You don't know the answer and a quick Google / Wikipedia search didn't make it clear either.
  • Two or more options look equally correct after looking it up.
  • The question requires specialist context you don't have and can't quickly acquire (e.g. niche fiqh details, obscure regional history).
  • You can also flag the question if the ambiguity is a problem with the question itself, not just your knowledge.

When to flag a question

Click the ⚑ flag button and add a short note. The cases below cover the most common reasons.

Flag vs edit — pick one rule of thumb:
  • Edit when you can fix it yourself — a typo, missing space, duplicate word, etc. No flag needed afterwards; just edit and move on.
  • Flag when you need the admin to look at it — wrong subdomain, no correct answer, ambiguity, missing visual, etc. These can't be fixed by changing the text.
  • Both at once is fine but rarely necessary. Don't flag a question just because you edited a typo in it.

Flagging is independent of answering. You can flag without picking an option, or pick and still flag. Both are useful signals.

When to edit (and when not to)

Click the ✎ edit button to fix small problems in the question or option text. Use it for:

  • Spelling / typos (e.g. اقبا → اقبال, پکستان → پاکستان).
  • Spacing fixes, missing diacritics, mangled Urdu/Arabic letter forms that you can correct from context.
  • Removing stray characters — extra punctuation, duplicated words, garbage left over from scraping.
  • Wrong subdomain — click the subdomain pill at the top of the MCQ and pick the correct one from the dropdown. This edit doesn't need the ✎ edit button to be active.

Don't translate technical English terms or proper names. HTML/CSS tags (<a>, <p>), scientific notation, brand names, English book titles, and similar content should stay in English. Translate only when the English is clearly a slip and there's an unambiguous Urdu equivalent (mango → آم, water → پانی, etc.).

Do not edit to:

  • Change the meaning of the question or an option.
  • Make a wrong answer into a right one.
  • Rewrite a poorly-worded question — flag it instead and let the admin handle it.

Submitting your work

  • Open the batch you've been working on, click Export.
  • A JSON file like YourHandle__batch_042.json downloads.
  • Upload it to the shared Drive folder: Urdu MMLU annotations (Google Drive)
  • You can re-export and re-upload any time; the admin's tool merges submissions and keeps the latest. The filename stays the same so re-uploads overwrite cleanly.
One JSON per batch. Don't merge multiple batches into one file.

Pacing

Aim for ~50 MCQs per sitting (one batch). At 15–30 seconds per question (including a quick search where useful) that's 15–25 minutes of focused work. Take a break after each batch; tired annotators miss things.

Accuracy > speed. It's fine if a batch takes 30 minutes instead of 15 because you fact-checked the tricky ones. We'd rather have a slower, correct annotation than a fast, wrong one.
